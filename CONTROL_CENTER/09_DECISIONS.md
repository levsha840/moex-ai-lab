## DECISION-001: Validation-first подход

**Дата:** Architecture Refresh

**Контекст:**
Проект мог развиваться в двух направлениях:
- Добавлять новые стратегии и торговые правила.
- Сначала построить механизм честной проверки.

**Решение:**
Приоритет отдаётся validation infrastructure (WalkForward + Cost Model) над новыми стратегиями.

**Причина:**
Стратегия без walk-forward валидации — это гипотеза, а не торговый инструмент. Система должна уметь отвергать плохие гипотезы. Именно это защищает от самообмана и потери капитала.

**Последствия:**
- Новые production-стратегии не добавляются до завершения v1.9.
- Demo-стратегии остаются в кодовой базе только как smoke-test.
- Порядок релизов: v1.8 (Allocation) → v1.9 (WalkForward + Cost) → v2.0 (Data Quality) → исследования.

## DECISION-002: RSI/SMA стратегии — только demo

**Дата:** Architecture Refresh

**Контекст:**
Стратегии `RSIOversoldNotDowntrend` и `TrendUpSMAConfirm` написаны для демонстрации API pipeline. Возникал вопрос: можно ли их использовать для paper trading.

**Решение:**
Эти стратегии официально признаны **Reference / Demo**. Они не являются кандидатами для paper или live trading.

**Причина:**
Классические RSI/SMA сигналы не имеют доказанного edge на MOEX при учёте реальных издержек. Их публичная известность означает, что любой возможный edge давно арбитрирован.

**Последствия:**
В кодовой базе вводится разделение через `source = "demo"` / `source = "research"` / `source = "production"` в метаданных `BaseStrategy`. Demo-стратегии никогда не запускаются в paper/live.

## DECISION-003: Portfolio Allocation Engine — минимальный scope

**Дата:** Architecture Refresh

**Контекст:**
v1.8 мог включать сложные методы аллокации: Kelly criterion, mean-variance, Sharpe-weighted.

**Решение:**
v1.8 реализует только: equal weight + max_position_pct + max_correlated_pct. Сложные методы — не в этом релизе.

**Причина:**
Сложные методы аллокации имеют собственный риск переобучения. До появления validated production-стратегий сложный аллокатор нечего аллоцировать. Простой аллокатор + RiskEngine уже дают достаточную защиту.

## DECISION-004: Data Quality Layer как prerequisite для исследований

**Дата:** Architecture Refresh

**Контекст:**
Планировалось начинать MOEX Hypothesis Lab (v2.2) параллельно с другими релизами.

**Решение:**
Data Quality Layer (v2.0) обязателен перед любыми исследованиями (v2.1+). Без него результаты бэктестов ненадёжны.

**Причина:**
Исторические данные MOEX содержат: неоткорректированные корпоративные действия, торговые приостановки, survivorship bias. Исследование на таких данных даёт ложные результаты.
