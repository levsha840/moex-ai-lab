# 05_SYSTEM_VISION

MOEX AI LAB — системное видение платформы после архитектурного обновления.

## Миссия

MOEX AI LAB строится не как одиночный торговый бот, а как платформа исполнения и сопровождения алгоритмических стратегий для MOEX.

Цель платформы — систематически находить, проверять, запускать, контролировать и отключать торговые стратегии в дисциплинированном инженерном процессе.

## Что платформа НЕ является (сейчас)

- Не готовая прибыльная торговая система.
- Не продукт для стороннего пользователя.
- Не алгоритм с доказанным edge.

Текущие стратегии (`RSIOversoldNotDowntrend`, `TrendUpSMAConfirm`, `ThresholdCloseStrategy`) — **демонстрационные**. Они показывают, как работает pipeline, а не как зарабатывать.

## Главный принцип

Любая стратегия считается непригодной, пока она не доказала обратное.

Стратегия допускается к капиталу только после прохождения полного lifecycle:

```text
Idea
  -> Research
  -> Implementation
  -> Backtest (с реалистичной моделью издержек)
  -> WalkForward (OOS-валидация)
  -> Paper Trading
  -> Production
  -> Continuous Supervision
  -> Scale / Disable / Archive
```

## Четыре контура платформы

### 1. Production Core

Отвечает за исполнение решений:

```text
StrategyEngine
  -> PortfolioAllocationEngine
  -> RiskEngine
  -> PaperTradingEngine / BrokerExecutionEngine
```

Требования: детерминированность, минимальные побочные эффекты, dependency injection, воспроизводимость решений, отсутствие зависимости от исследовательского кода.

### 2. Validation Core

Отвечает за допуск стратегий:

```text
Backtest
  -> Cost Model (спред, T+2, налог 13%, рыночное влияние)
  -> WalkForward
  -> OOS Metrics
  -> PASS / FAIL
```

Главная задача Validation Core — чаще говорить `FAIL`, чем `PASS`.

### 3. Research Core

Отвечает за генерацию и проверку гипотез:

- hypothesis registry;
- feature engineering;
- statistical validation;
- strategy discovery;
- будущий AI-assisted research.

Research Core не имеет прямого доступа к реальному исполнению сделок.

### 4. Operations Core

Отвечает за сопровождение работающих стратегий:

- strategy supervisor;
- деградация метрик;
- drawdown control;
- regime drift;
- audit trail;
- decision logs.

## Strategy Lifecycle

Стратегия — управляемая сущность, а не просто класс в коде.

Статусы: `research` -> `backtest` -> `walkforward` -> `paper_trading` -> `production_candidate` -> `production` -> `disabled` -> `archived`.

## Критерии допуска к paper trading

- Sharpe ratio > 0.8 на каждом OOS-окне walk-forward.
- Max drawdown < 15% на OOS.
- Положительный результат после всех издержек (комиссия + спред + налог 13%).
- Стратегия работает в двух или более рыночных режимах.
- Минимум 100 сделок в OOS-выборке (статистическая значимость).

## Целевые рынки и таймфреймы

- **Рынок:** MOEX, российские акции (liquid mid/large cap).
- **Таймфреймы:** дневной (D1) как приоритет для исследований; внутридневной (M5–H1) как второй этап.
- **Горизонт позиции:** swing (3–20 дней) как основной.

## Главный KPI платформы

На раннем этапе главным KPI является не доходность, а скорость и качество проверки:

- сколько гипотез проверено;
- сколько отклонено;
- сколько прошло walk-forward;
- сколько прошло paper trading;
- сколько стратегий отключено корректно;
- сколько времени занимает путь от идеи до объективного вердикта.

## Правило развития

Каждый новый модуль должен либо:

1. улучшать способность платформы зарабатывать;
2. улучшать способность платформы отличать рабочие стратегии от нерабочих;
3. снижать риск самообмана, переобучения или операционных ошибок.

Если модуль не делает ничего из этого — его необходимость должна быть поставлена под сомнение.
