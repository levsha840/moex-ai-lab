# 10_ARCHITECTURE_DECISIONS

MOEX AI LAB — журнал архитектурных решений (ADR).

Каждое решение фиксирует: контекст, решение, причину, последствия.

---

## ADR-001: Детерминированные движки без доступа к БД

**Статус:** принято (v1.3)

**Контекст:**
Торговые движки (ReplayEngine, StrategyEngine, PaperTradingEngine, RiskEngine) могли бы писать в БД напрямую.

**Решение:**
Все доменные движки — pure объекты без side-эффектов. Они не знают о базе данных.

**Причина:**
- Детерминизм: одни и те же входные данные всегда дают одинаковый результат.
- Тестируемость: тесты не требуют БД или моков.
- Изоляция: доменная логика не зависит от инфраструктуры.

**Последствия:**
Persistence вынесен в отдельный слой (`core/persistence`). Движки получают данные через параметры, результаты возвращают через объекты.

## ADR-002: Protocol-based Persistence Layer

**Статус:** принято (v1.6.1)

**Контекст:**
PositionManager изначально хранил позиции в dict. Нужно было добавить PostgreSQL backend.

**Решение:**
Введён `PositionRepository` как `Protocol` (structural subtyping). `PersistenceFactory` выбирает backend. Бизнес-логика работает только через интерфейс.

**Причина:**
- Замена backend без изменения бизнес-логики.
- In-memory backend позволяет тестировать без БД.
- Открыто для Redis, ClickHouse в будущем.

**Последствия:**
PostgreSQL backend пока — заглушка. Это осознанный компромисс: не добавлять незавершённую реализацию в прод-код.

## ADR-003: RiskEngine — опциональная интеграция

**Статус:** принято (v1.7)

**Контекст:**
RiskEngine нужно было встроить в PaperTradingEngine. Возможны два подхода: обязательный (всегда нужен) или опциональный.

**Решение:**
`PaperTradingEngine(config, risk_engine=None)` — Risk Engine опционален. При отсутствии поведение не меняется.

**Причина:**
- Обратная совместимость: 6 существующих тестов не сломаны.
- Гибкость: можно запускать engine в тестах без риск-проверок.
- Staging: можно включать RiskEngine постепенно.

**Последствия:**
Возможна ситуация, когда PaperTradingEngine работает без защиты. Это допустимо для тестов и smoke-checks, но недопустимо для production paper trading.

## ADR-004: WalkForward Engine и Cost Model — единый релиз

**Статус:** принято (Architecture Refresh)

**Контекст:**
Изначально планировалось реализовать WalkForward Engine (v1.9) и Realistic Cost Model (v2.0) как отдельные релизы.

**Решение:**
WalkForward Engine и Realistic Cost Model реализуются в одном релизе (v1.9).

**Причина:**
WalkForward без честной модели издержек — это всё равно biased backtesting. Критерий "стратегия прошла проверку" не может быть определён без cost model. Разделение создаёт ложное чувство завершённости.

**Последствия:**
v1.9 — более крупный релиз. Cost Model должна включать: спред, рыночное влияние, T+2 settlement, налог 13%.

## ADR-005: RSI/SMA стратегии — только demo

**Статус:** принято (Architecture Refresh)

**Контекст:**
`RSIOversoldNotDowntrend`, `TrendUpSMAConfirm`, `ThresholdCloseStrategy` написаны для демонстрации API и smoke-test pipeline.

**Решение:**
Эти стратегии официально признаны **Reference / Demo** и не являются кандидатами для paper или live trading.

**Причина:**
- Сигналы публично известны 20+ лет, edge давно арбитрирован.
- На MOEX при комиссии 0.05–0.3% за сделку простые RSI/SMA стратегии не приносят дохода.
- Их присутствие в кодовой базе нужно для CI/CD и документации API, не для торговли.

**Последствия:**
В кодовой базе необходимо явно отделять demo-стратегии от production-кандидатов (через `source = "demo"` в метаданных `BaseStrategy`).

## ADR-006: Data Quality Layer до исследований

**Статус:** принято (Architecture Refresh)

**Контекст:**
WalkForward Engine будет использовать исторические данные MOEX. Эти данные содержат: сплиты без корректировки, дивидендные отсечки, торговые приостановки (февраль 2022), делистинги.

**Решение:**
Data Quality Layer (v2.0) реализуется до начала серьёзных исследований (v2.1+). Без него результаты бэктестов ненадёжны.

**Причина:**
- Survivorship bias: тестирование только на существующих бумагах завышает результат.
- Корпоративные действия без корректировки создают ложные сигналы (например, "падение" на 30% в день дивидендной отсечки).
- Приостановки торгов нельзя считать обычными свечами.

**Последствия:**
v2.0 — обязательный релиз перед v2.1 (Market Regime) и v2.2 (Hypothesis Lab).

## ADR-007: Два параллельных трекера позиций — известная проблема

**Статус:** открыто, решение не принято

**Контекст:**
`PaperTradingEngine` ведёт собственный учёт позиций через `PaperPosition`. `PositionManager` ведёт отдельный учёт через `Position`. Они не связаны.

**Варианты:**
1. `PaperTradingEngine` делегирует позиции в `PositionManager`.
2. Оставить как есть: `PaperPosition` — исполнение, `Position` — управление.
3. Унифицировать модели в одну доменную сущность.

**Статус:**
Решение требует участия архитектора проекта. Код не менять до принятия решения.

## Правило

ADR добавляется при каждом нетривиальном архитектурном выборе. Каждый ADR имеет статус: `принято`, `отклонено`, `открыто`, `устарело`.
