# 08_PROJECT_RULES

MOEX AI LAB — правила разработки проекта.

---

## Правила разработки

### Перед началом любого релиза

1. `git status` — рабочее дерево чистое.
2. `git pull` — локальная версия совпадает с remote.
3. `pytest` — все тесты проходят.
4. Если любое из трёх не выполнено — работа не начинается.

### Во время разработки

- Минимальные изменения: только то, что нужно для текущей задачи.
- Архитектуру не менять без явного решения в `10_ARCHITECTURE_DECISIONS.md`.
- Не изменять `.env`, секреты, API-ключи без явного запроса.
- Не выполнять `git push` без подтверждения.

### Стратегии

- Demo-стратегии (`source = "demo"`) не допускаются к paper/live trading.
- Новая production-стратегия добавляется только после прохождения walk-forward (v1.9+).
- `RSIOversoldNotDowntrend`, `TrendUpSMAConfirm`, `ThresholdCloseStrategy` — demo. Не изменять их статус.

---

## Правила тестирования

- Каждый новый модуль сопровождается тестами.
- Тесты не используют реальную БД — только in-memory backend или моки.
- Регрессии недопустимы: новый код не ломает существующие тесты.
- `tests/persistence/` и любые новые поддиректории тестов требуют `__init__.py`.

---

## Правила архитектуры

- Доменные движки (`ReplayEngine`, `StrategyEngine`, `PaperTradingEngine`, `RiskEngine`) не имеют доступа к БД.
- Бизнес-логика работает только через интерфейсы Persistence Layer.
- Новые persistence backend реализуются полностью или не добавляются вообще (не заглушки поверх заглушек).
- `core/portfolio/` (старый слой) — не развивать до принятия ADR-007.

---

## Правила документации

- После завершения каждого релиза обновляются все документы CONTROL_CENTER.
- CONTROL_CENTER — единственный источник актуального состояния проекта.
- Архитектурные решения фиксируются в `10_ARCHITECTURE_DECISIONS.md` до начала реализации.
- Расхождение между документацией и кодом — баг, исправляется в том же коммите.

---

## Процесс релиза

1. Запустить pre-flight: `git status`, `git pull`, `git log`, `pytest`.
2. Реализовать изменения.
3. Запустить `pytest` — все тесты зелёные.
4. Обновить CONTROL_CENTER.
5. `git add .` → `git commit -m "vX.Y Name"` → `git push`.
6. Проверить: `git status` показывает `working tree clean`.

---

```
vX.Y Release Name        # релизный коммит
fix: краткое описание    # исправление бага
docs: краткое описание   # только документация
```
